# 多智能体竞技 `/arena`

> 3 个 agent 并行解决同一个问题，评审模型选出最佳方案。全网独一份。

## 快速开始

```
/arena 实现一个限流器
/arena write a rate limiter function
/arena 添加用户注册API
```

## 工作流

```
/arena 任务描述
       ↓
  🟢 Agent A: "最简方案，先正确再优化"     ──→ 独立实现
  🔵 Agent B: "生产级，边界处理从头开始"    ──→ 独立实现（并行）
  🟣 Agent C: "创意方案，寻找优雅模式"      ──→ 独立实现
       ↓
  各自 12 步上限，完整工具链（read/edit/bash/verify）
       ↓
  评审模型（glm-5.2）打分：
    正确性 / 代码质量 / 完整性 / 效率
       ↓
  🏆 胜出方案展示 + 评审理由
```

## 三种策略

| Agent | 策略 | 擅长 |
|---|---|---|
| 🟢 Approach A | 最简单可读的实现 | 可读性 + 正确性 |
| 🔵 Approach B | 生产级，边界处理 | 可靠性 + 完整性 |
| 🟣 Approach C | 创意/非常规方案 | 优雅 + 创新 |

## 评审标准

评审模型按四个维度打分（0-100）：

1. **正确性**：是否真正解决了问题？边界情况处理了吗？
2. **代码质量**：是否整洁、可读、可维护？是否匹配现有模式？
3. **完整性**：有测试吗？错误处理？文档？
4. **效率**：性能好吗？有不必要的复杂度吗？

两个方案同等时，**选更简单的那个**。

## 使用场景

- **算法实现**：不同时间/空间复杂度的方案对比
- **架构设计**：微服务 vs 单体 vs 混合，选最合适的
- **Bug 修复**：补丁修复 vs 根因修复 vs 架构调整，选最稳的
- **新功能**：REST vs GraphQL vs gRPC，选最合适的
