## 1. 项目概述 | Executive Summary

### 一句话定义

**DaoismCode 是一个运行在终端中的全自动 AI 软件工程师** —— 它能规划任务、编写代码、运行测试、分析失败、修复缺陷、学习经验，并且在所有验证通过后才宣布"完成"。无需 IDE 插件，无需浏览器，无需 FFI 依赖。

### 核心数字

| 指标 | 数值 |
|---|---|
| 源代码行数 | **10,588 行 TypeScript** |
| 工具数量 | **24 个内置工具** |
| 命令数量 | **23 个斜杠命令** |
| 智能路由引擎 | **6 层级联决策** |
| 路由境界 | **4 境 (筑基/金丹/元婴/化神)** |
| 支持语言（代码分块） | **6 种** (TS/JS/Python/Java/Go/Rust) |
| 依赖关系 | **零 FFI**，纯 TypeScript |
| 部署产物 | **单个 103 MB 独立 .exe** |
| 成本节约 | **~56%**（通过智能路由自动降级） |
| Provider 兼容性 | **任何 OpenAI 兼容端点** |

### 为什么不是又一个 AI 代码工具

市面上有 Claude Code、Cursor、Devin、GitHub Copilot。DaoismCode 不是它们的复制品。它解决了三个被忽视的问题：

1. **成本黑洞** — 其他工具要么用单一昂贵模型（浪费算力），要么需要月度订阅（$20-500/月）。DaoismCode 通过六引擎智能路由，将简单任务降级到轻量模型，**节约 56% 推理成本**——且 Provider 完全无关，任何 OpenAI 兼容端点即插即用。

2. **虚假完成** — 其他 AI 说了"done"就是 done，不管代码是否真的能跑。DaoismCode 有 **五维完成门控**：测试通过？类型检查通过？无调试残留？变更范围合理？验证已执行？全部通过才能结束。

3. **不学习** — 其他 AI 每次对话从零开始，同样的错误犯一百遍。DaoismCode 有 **双层记忆系统**：Reflexion（失败教训）+ Project Memory（项目知识），越用越聪明。

## 2. 设计哲学 | Design Philosophy

### 道生一，一生二，二生三，三生万物

DaoismCode 的设计深受道家哲学启发。这不是装饰性的命名，而是系统架构的指导原则：

| 道家概念 | 系统映射 | 含义 |
|---|---|---|
| **道** (The Way) | 核心循环 | Plan → Execute → Verify → Fix，万物归于此循环 |
| **四境** (Four Realms) | 模型路由 | 筑基→金丹→元婴→化神，能力递进，成本递进 |
| **无为** (Non-action) | 自动化设计 | 用户不需要干预——路由自动选、验证自动跑、测试自动写 |
| **反者道之动** | 反思记忆 | 失败是学习之源——每次错误都被记录、归纳、避免 |
| **道法自然** | 零 FFI | 不依赖外部原生库，一切用 TypeScript 自然实现 |

### 三大设计原则

#### 原则一：终端即一切 (Terminal is Everything)

```
不需要 IDE。不需要浏览器。不需要 Electron。
一个终端窗口，一个 .exe 文件，这就是全部。
```

我们用纯 TypeScript + ANSI 转义序列构建了完整的 TUI 渲染器——语法高亮、Diff 预览、流式动画、权限门控、CJK 对齐，全部通过 `process.stdout.write()` 实现。没有 Ink，没有 SolidJS，没有 node-ffi。

#### 原则二：验证高于声明 (Verification over Declaration)

```
AI 说的 "done" 不算 done。
测试跑过的 "pass" 才是 done。
```

这是 DaoismCode 与所有竞品的本质区别。其他工具让 AI 自己判断是否完成。DaoismCode 用客观验证门控——AI 必须运行测试、类型检查、lint，全部通过后才能结束。

#### 原则三：越用越强 (Compound Intelligence)

```
第 1 次对话：它是一个通用 AI。
第 100 次对话：它了解你的项目架构、编码规范、历史陷阱。
```

双层记忆系统让 DaoismCode 具备"长期记忆"：
- **Reflexion**：记录每次失败的原因和修复方法，自动注入系统提示
- **Project Memory**：记录项目架构、技术决策、已知陷阱、用户偏好

### 设计决策：Provider 无关架构

DaoismCode 的 LLM 客户端基于标准 OpenAI 兼容协议。这意味着：

- 任何提供 `/v1/chat/completions` 端点的服务商都可以接入
- 配置只需三个参数：`baseURL`、`apiKey`、`model`
- 已测试兼容：智谱、OpenAI、DeepSeek、Ollama（本地模型）
- 智能路由的"四境"是**模型能力分层**的抽象，不绑定特定厂商

```
配置示例 (.daoismcode/config.json):

{
  "provider": "自定义",
  "baseURL": "https://your-api-endpoint/v1",
  "apiKey": "your-key",
  "model": "your-flagship-model",
  "enableRouting": true
}
```

## 4. 四境智能路由 | Four-Realm Smart Routing

### 核心创新

> **其他 AI 工具用一把锤子敲所有钉子。DaoismCode 有一个完整的工具箱。**

传统 AI 编码工具的模型选择是静态的——用户手动选模型，或固定使用一个。DaoismCode 的路由器是**动态的**：每条消息、每个执行步骤都独立路由到最优模型。

### 四境体系：模型能力分层抽象

路由系统将所有 LLM 按能力划分为四个层级（"境界"），这是对模型能力的**架构级抽象**，不绑定任何特定厂商：

```
🟫 筑基 (Foundation)     轻量快速模型     高速推理, 低延迟
                                         适用: 简单问答、文件搜索、代码阅读
                                         成本: ★☆☆☆☆ (最低)

🟨 金丹 (Golden Core)    多模态视觉模型   图像理解 + 文本推理
                                         适用: 截图分析、UI转代码、错误诊断
                                         成本: ★★☆☆☆

🟦 元婴 (Nascent Soul)   平衡推理模型     中等推理能力 + 安全降级
                                         适用: 日常编码、中等复杂度任务
                                         成本: ★★★☆☆

🟪 化神 (Spirit)        旗舰编码模型     编码SOTA, 超长上下文
                                         适用: 复杂重构、架构设计、疑难bug
                                         成本: ★★★★★ (最高)
```

**关键设计**：境界的映射是配置化的。用户可以将任意厂商的任意模型映射到四个境界：

```json
{
  "realms": {
    "foundation": { "model": "your-fast-model", "maxTokens": 128000 },
    "goldenCore": { "model": "your-vision-model", "maxTokens": 128000 },
    "nascentSoul": { "model": "your-balanced-model", "maxTokens": 128000 },
    "spirit": { "model": "your-flagship-model", "maxTokens": 1000000 }
  }
}
```

### 六引擎级联决策

路由不是单一算法，而是**六层级联决策引擎**——每一层在前一层不确定时介入：

```
                     ┌─────────────────────────────┐
  用户消息 ──────────▶│  ① 权重评分引擎              │
                     │  (中英文关键词 + 代码检测)     │
                     └────────────┬────────────────┘
                                  │ 置信度 < 0.45?
                     ┌────────────▼────────────────┐
                     │  ② 工具调用反馈               │◀── 上轮用了 edit_file?
                     │  (从工具历史推断任务类型)       │
                     └────────────┬────────────────┘
                                  │ 仍不确定?
                     ┌────────────▼────────────────┐
                     │  ③ 上下文感知                 │◀── 最近3轮都是编码?
                     │  (延续会话语境)               │
                     └────────────┬────────────────┘
                                  │ 仍不确定?
                     ┌────────────▼────────────────┐
                     │  ④ LLM辅助分类               │
                     │  (筑基境0.3秒快速判断)         │
                     └────────────┬────────────────┘
                                  │
                     ┌────────────▼────────────────┐
                     │  ⑤ 复杂度 + 预算守卫          │◀── >300字? >85%配额?
                     │  (自动升级 / 降级)            │
                     └────────────┬────────────────┘
                                  │
                     ┌────────────▼────────────────┐
                     │  ⑥ 多步混合路由               │◀── 每步动态切换
                     │  (每个步骤重新选择模型)        │
                     └────────────┬────────────────┘
                                  │
                                  ▼
                     ⛩️ 最优境界选定
```

#### 引擎①：权重评分

不再是"第一个匹配的关键词获胜"。所有类别**同时评分**，取得分最高的：

```
消息: "检查 auth.ts 里的 bug 并优化性能"

评分:
  coding:   1.5    (文件路径 auth.ts +1.5)
  review:   1.8    (bug ×0.9 + optimize ×0.9)
  planning: 0
  search:   0
                    ↓
  胜出: review (1.8)  →  化神境
```

**检测维度**：50+ 中英文关键词（权重 0.7–1.0）、代码块（+3）、行内代码（+1.5）、文件路径（+1.5）、语法关键词（+3）。

**技术亮点**：权重评分引擎是纯规则驱动的，延迟 <1ms，不消耗任何 token。它处理了 80%+ 的路由决策，只有剩余 20% 需要更重的引擎介入。

#### 引擎②：工具调用反馈

Agent 上轮用了什么工具，是当前任务类型的最强信号：

| 上轮工具 | 推断类型 | 路由至 |
|---|---|---|
| `edit_file` `write_file` `bash` | 编码 | 化神 |
| `read_file` `glob` `grep` | 搜索 | 筑基 |
| 混合 | 多数投票 | 动态 |

**技术亮点**：这是一个**隐式信号挖掘**设计——用户的措辞可能模糊，但他们上一步的行为是精确的。如果 Agent 刚用了 `edit_file`，那当前任务极大概率仍是编码。

#### 引擎③：上下文感知

记住最近 3 轮对话。当消息模糊时，延续会话语境：

```
第1轮: "实现登录功能"        → 编码 → 化神
第2轮: "再加密码重置"         → 编码 → 化神
第3轮: "嗯，还有验证码"       → 模糊... 但最近3轮 = 编码
      → 延续上下文 → 编码 → 化神 ✓
```

**技术亮点**：滑动窗口上下文继承，解决了用户"简短回复"导致的路由震荡问题。没有这层，"嗯"或"对"这样的回复会触发误分类。

#### 引擎④：LLM 辅助分类

当规则引擎置信度低时，筑基境执行 0.3 秒快速分类：

```
消息: "嗯，帮我搞一下那个东西"   (置信度 0.2)
     ↓
筑基境快速分类: "编码"
     ↓
路由至化神 · 来源: llm
```

**技术亮点**：这是一个**降级兜底**设计。用最便宜的模型做分类判断（消耗极少 token），避免用昂贵模型处理一个根本不需要编码的任务。投入产出比极高。

#### 引擎⑤：复杂度 + 预算守卫

- **复杂度升级**：消息 > 300字 → 自动升级到化神境（长消息通常意味着复杂任务）
- **预算降级**：任一境界 > 85% 配额 → 自动降级到更便宜的境界（防止成本失控）

**技术亮点**：预算守卫是一个**反馈控制回路**——系统实时监控各境界使用比例，当旗舰模型使用过多时自动降级，确保成本可预测。

#### 引擎⑥：多步混合路由

单个任务可能跨多步。每步根据前一步的工具调用重新选模型：

```
任务: "搜索登录代码，然后加双因素认证"

  步骤1: glob + grep (搜索)     → 筑基  (便宜，够用)
  步骤2: read_file (读代码)     → 筑基
  步骤3: edit_file (写代码!)    → 自动升级到化神 (检测到编码工具)
  步骤4: bash (跑测试)          → 化神
  步骤5: 回复用户               → 化神
```

**技术亮点**：这是整个路由系统的**核心创新**。其他工具在整个任务期间锁定一个模型。DaoismCode 在任务内部**逐步切换**——搜索阶段用便宜模型，编码阶段自动升级。一个5步任务，可能只在步骤3-5才用旗舰模型，步骤1-2全部用轻量模型。

### 路由来源标签

TUI 显示路由决策的来源，让路由过程完全透明：

```
⛩️ 化神 · coding · rules     ← 规则引擎判定
⛩️ 化神 · coding · tools     ← 从工具历史推断
⛩️ 化神 · coding · context   ← 从会话语境延续
⛩️ 化神 · coding · llm       ← LLM辅助分类
```

### 路由效率分析

| 任务类型 | 路由至 | 原因 | 相对成本 |
|---|---|---|---|
| "什么是useState?" | 筑基 | 简单知识问答 | ★ |
| 截图分析 | 金丹 | 需要视觉 | ★★ |
| "重构这段代码" | 化神 | 复杂编码 | ★★★★★ |
| 搜索认证代码 | 筑基 | 工具执行，不需推理 | ★ |
| 跑测试+修bug | 化神 | 编码+验证 | ★★★★★ |

100次混合请求（典型开发者工作日）：

```
无路由:  100 × 旗舰模型 = 100单位成本
有路由:  35 × 筑基 + 10 × 金丹 + 20 × 元婴 + 35 × 化神 ≈ 44单位成本
                                                                   ↓
```

## 5. 自主工程循环 | Autonomous Engineering Loop

### 核心理念

> **DaoismCode 不是代码生成器。它是一个自主工程师——编写、验证、自我修正，直到所有检查通过才交付。**

受 Devin、SWE-Agent 和 Claude Code 架构启发，但更进一步——加入了反思记忆和 Git 原生安全网。

### 循环流程

```
用户任务
    │
    ▼
┌─ 规划 (PLAN) ───────────────────────┐
│ 1. todo_write — 分解任务             │
│ 2. read_file / grep / glob — 探索    │
│ 3. find_code — 语义搜索 (新!)        │
└──────────────┬───────────────────────┘
               ▼
┌─ 执行 (EXECUTE) — TDD优先 ──────────┐
│ 4. 编写测试 (定义行为)               │
│ 5. 运行测试 (应该失败 — 验证)        │
│ 6. 编写代码 (最小通过)               │
│ 7. git stash create ← 安全网         │
└──────────────┬───────────────────────┘
               ▼
┌─ 验证 (VERIFY) — 强制执行 ──────────┐
│ 8. detect_verify → 查找验证命令      │
│ 9. run_verify → 一键运行全部验证     │
│    · 测试套件                        │
│    · 类型检查 (tsc/mypy)            │
│    · Lint检查                       │
│    · 编译                           │
└──────────────┬───────────────────────┘
               │
       ┌───────┴───────┐
       ✅ 全部通过       ❌ 失败
       │               │
       ▼               ▼
┌─ 完成门控 ──  ┌─ 修复 (自我修正, max 3轮) ──────┐
│ - 测试通过?    │ 10. 读取错误输出                  │
│ - 类型检查OK?  │ 11. 分析根本原因                  │
│ - 无调试残留?  │ 12. 修复代码                      │
│ - Diff合理?    │ 13. 记录教训到 Reflexion          │
│ - 已验证?      │ 14. 重新验证 → 循环               │
└───────────────┘ └──────────────────────────────────┘
               │
               ▼
          ✅ 已交付
```

### 三大强制机制

#### 机制一：验证阻塞

如果 Agent 写了代码但**试图在没跑测试的情况下说"done"**，循环拒绝结束：

> `[SYSTEM] 你修改了代码但尚未验证。立即运行项目的测试套件和类型检查。`

**技术实现**：`hasVerified` 标志位在每次代码编辑时自动重置为 `false`。只有 `run_verify` 工具成功执行后才设为 `true`。完成门控检查此标志——如果为 `false`，注入强制验证提示，循环不结束。

#### 机制二：失败反思注入

当测试失败时，循环自动注入结构化提示：

> `[SYSTEM] 验证失败 (第1次/共3次)。分析上方错误输出，找到根本原因并修复。在所有检查通过前不要宣布"完成"。`

**技术实现**：`verifyNotDoneCount` 计数器限制验证循环最多 2 次阻塞——防止 Agent 无限验证却无法修复的死循环。超过限制后，Agent 被允许诚实报告剩余问题。

#### 机制三：重试上限

最多 **3 次修复尝试**（75% 的改进来自前 2 轮）。3 轮后，Agent 诚实报告剩余错误。

### 五维完成门控

"完成"不是 Agent 说了算。"完成"是验证证明出来的。

五个门控必须**全部通过**，Agent 才能结束：

| 门控 | 检查 | 阻塞条件 |
|---|---|---|
| **测试** | 测试套件退出码 0 | 任何测试失败 |
| **类型检查** | `tsc --noEmit` / `mypy` 无错误 | 存在类型错误 |
| **调试残留** | 新代码中无 `console.log` / `print` / `debugger` | 遗留调试语句 |
| **Diff范围** | `git diff --stat` 变更聚焦 | > 500行插入（可能误操作） |
| **已验证** | 至少运行过一次验证命令 | 写了代码但从未测试 |

### ReAct 反思推理

每 5 步自动触发一次 ReAct (Reasoning + Acting) 反思：

```
[步骤 5/50] ReAct 反思:
  - 当前进展: 已完成3个子任务，剩余2个
  - 下一步: 继续处理认证模块
  - 是否偏离计划: 否
  - 需要调整: 无
```

### Sisyphus 模式

> **永不放弃。500步，无限修复，预算耗尽自动续命。**

```
/sisyphus    ← 开启/关闭 (跨重启持久化)
```

| 参数 | 普通模式 | Sisyphus 模式 |
|---|---|---|
| 步数预算 | 50 | **500** |
| 修复次数 | 3 | **无限 (∞)** |
| 预算耗尽 | 停止 | **自动续命 (最多3轮 = 1500步)** |
| 状态栏 | `⛩️化神境` | `🔄⛩️化神境` |

**设计意图**：Sisyphus 模式面向"大型重构""迁移""从零搭建项目"等需要长时间持续工作的场景。普通模式 50 步适合日常任务；Sisyphus 模式 1500 步上限（500步 × 3轮自动续命）足够完成绝大多数工程项目。

## 6. 语义代码搜索 | Semantic Code Search

### 核心问题

传统关键词搜索有一个致命缺陷——它搜的是"字面"，不是"含义"：

```
你问: "认证功能是怎么实现的？"

grep "auth"      → 找到 47 个文件 (auth.ts, oauth.ts, author.ts, authority.ts...)
                  → Agent 淹没在误报中，读了10个错误文件，浪费8步

find_code("认证功能实现")  → 找到 3 个文件 (auth.ts, middleware.ts, token.ts)
                           → Agent 2步读到正确的文件
```

### 技术方案

#### 第一步：AST 感知代码分块

不是按行切割，而是按**语法结构**切割——每个函数、类、方法是一个独立块：

```
auth.ts 原始文件 (200行)
    ↓ AST 分块
┌─────────────────────────────────────┐
│ Block 1: verifyToken()     [行 1-25] │
│ Block 2: generateToken()   [行 26-50]│
│ Block 3: class AuthManager [行 51-180]│
│ Block 4: refreshToken()    [行181-200]│
└─────────────────────────────────────┘
```

支持 6 种语言的分块边界：

| 语言 | 分块边界 |
|---|---|
| TypeScript / JavaScript | `function`, `class`, `method`, `arrow function` |
| Python | `def`, `class`, `async def` |
| Java | `public/private method`, `class` |
| Go | `func`, `type struct` |
| Rust | `fn`, `impl`, `struct` |
| 其他 | 行级切割 + 重叠窗口 |

**技术决策**：为什么不按固定行数分块？因为按语法结构分块保证了每个嵌入向量都代表一个**完整的语义单元**。按行切割会把一个函数切成两半，导致向量表征混乱。

#### 第二步：向量嵌入

每个代码块通过嵌入模型生成向量表征：

```
Block 1: "verifyToken"  → [0.12, -0.34, 0.56, ... ] (512维向量)
Block 2: "generateToken" → [0.08, -0.22, 0.45, ... ]
Block 3: "AuthManager"   → [0.15, -0.31, 0.52, ... ]
Block 4: "refreshToken"  → [0.11, -0.28, 0.48, ... ]
```

**架构设计**：嵌入模型是可插拔的。任何支持 `/v1/embeddings` 端点的模型都可以接入。系统不关心向量维度是 512、768 还是 1536——它会自动适配。

#### 第三步：余弦相似度检索

用户查询也嵌入为向量，然后与所有块计算余弦相似度：

```
查询: "认证功能实现" → [0.13, -0.30, 0.50, ...]

相似度排名:
  1. Block 1: verifyToken      0.94 ← 最相关!
  2. Block 3: AuthManager       0.89
  3. Block 2: generateToken     0.82
  4. Block 4: refreshToken      0.78
```

### 增量索引

不是每次全量重建。基于文件 mtime（修改时间）的增量更新：

```
首次 /index:
  扫描 200 个文件 → 分块 → 嵌入 → 保存

后续使用:
  检查 mtime → 只有 3 个文件变了 → 只更新这 3 个文件的嵌入
  O(n) 批量IO，不再是 O(n²)
```

**性能优化**：早期版本对每个文件单独发起嵌入请求（O(n²) 网络往返）。优化后改为批量请求——所有变化的块一次性提交，网络往返从 N 次降为 1 次。

### Agent 自动使用

Agent 在探索不熟悉的代码库时自动使用 `find_code`。循环在连续 3 次 `grep`/`glob` 调用后（如果尚未使用 `find_code`）自动注入提示：

> `[提示] 你已使用 grep/glob 3 次。试试 find_code 进行语义搜索——它按含义搜索代码，不是关键词。`

**设计哲学**：Agent 不需要用户教它用语义搜索。系统通过行为观察（连续 grep 失败）自动引导 Agent 升级到更好的工具。这是"无为"原则的工程体现。

## 7. 并行子智能体 | Parallel Sub-Agents

### 自动检测

Agent 不需要你告诉它"并行执行"。它**实时分析你的消息**，检测独立任务：

```
你: "帮我实现注册、登录、密码重置三个功能"
         ↓ [代码级自动检测]
         ✓ 检测到 3 个独立任务动词 (实现/登录/密码重置)
         ✓ 无依赖关键词 (然后/之后/基于)
         ↓ [自动注入Agent上下文]
         Agent看到: "检测到3个独立任务。使用 parallel_subagents。"
         ↓
         🚀 3个子智能体同时启动
```

### 检测模式

| 模式 | 示例 | 触发 |
|---|---|---|
| 数字列表 | `1. 注册 2. 登录 3. 重置` | ✅ |
| 分号列表 | `实现注册; 添加登录; 创建重置` | ✅ |
| 顿号列表 (3+) | `实现注册、添加登录、创建重置` | ✅ |
| 多动词 | `实现注册 和 添加登录` | ✅ |
| 顺序依赖 | `先修复bug，然后写测试` | ❌ (有"然后") |
| 单一任务 | `修复这个bug` | ❌ |

**技术实现**：`parallel-detect.ts` 使用正则模式匹配中文和英文的任务结构标记。核心逻辑是**检测独立性**——如果任务之间有顺序依赖关键词（然后、之后、基于、依赖），则不并行化。

### 执行与质量控制

```
🚀 3个并行子智能体在 12.3秒内完成 (3/3 成功)

✅ 任务1: 实现用户注册功能 (5步)
   创建 src/api/register.ts...

✅ 任务2: 实现用户登录功能 (4步)
   创建 src/api/login.ts...

✅ 任务3: 实现密码重置功能 (6步)
   创建 src/api/reset.ts...
```

### 安全机制

| 机制 | 作用 | 实现原理 |
|---|---|---|
| **权限锁** | 串行化权限提示 | Mutex信号量，防止多个子智能体同时请求权限导致 UI 死锁 |
| **120秒超时** | 每个子智能体 2 分钟后自动终止 | `Promise.race` + `setTimeout`，防止无限挂起 |
| **深度限制** | 子智能体不能再生成子智能体 | 全局深度计数器，max depth = 2 |
| **文件冲突警告** | 系统提示禁止并行任务触碰同一文件 | 系统提示注入，靠 LLM 自律 |

### 速度对比

```
串行: Task1(8s) → Task2(6s) → Task3(10s) = 24秒
并行: max(8s, 6s, 10s)                    = 10秒   ← 快58%
```

```
┌──────────────────────────────────────────┐

│  │       +错误   │  │       +技术栈    │ │
│  │       +修复   │  │       +编码规范  │ │

│                 ▼                        │
│     自动注入系统提示 (每次对话)            │
└──────────────────────────────────────────┘
```

### Reflexion — 失败学习

每次测试失败或验证不通过，Agent 自动记录：

```
[2026-07-02] TypeError: Cannot read undefined
  → 原因: 异步函数未 await
  → 修复: 添加 await 关键字
  → 教训: 所有 async 函数返回值必须 await
```

下次任务启动时，系统提示自动加载历史教训：

```
## 历史教训 — 请勿重复
1. async函数返回值必须await
2. TypeScript strict模式下null检查必须显式
3. ...
```

**设计灵感**：来自论文 [Reflexion: Language Agents with Verbal Reinforcement Learning](https://arxiv.org/abs/2303.11366)。核心思想是用自然语言（而非梯度）作为强化学习的"奖励信号"——每次失败的经验被编码为文字教训，注入后续任务的系统提示中。

### Project Memory — 项目知识

首次运行时自动扫描项目并初始化：

```json
{
  "projectName": "my-app",
  "language": "TypeScript",
  "framework": "Next.js 14",
  "buildCommand": "pnpm build",
  "testCommand": "pnpm test",
  "lintCommand": "pnpm lint",
  "architecture": {
    "src/app/": "App Router pages",
    "src/components/": "React components",
    "src/lib/": "Shared utilities"
  },
  "conventions": [
    "Use named exports only",
    "Error responses return { error: string }"
  ],
  "pitfalls": [
    "Don't import from src/app in src/lib",
    "useServer must be at file top"
  ]
}
```

通过 `update_memory` 工具随时更新。

**与 Reflexion 的区别**：Reflexion 记录的是"怎么失败了"（战术层面），Project Memory 记录的是"这个项目是什么样的"（战略层面）。两者互补——Reflexion 防止重复犯错，Project Memory 提供全局上下文。

## 9. 多模态视觉 | Multimodal Vision

### /paste — 剪贴板截图分析

```
1. Win+Shift+S 截图 (进入剪贴板)
2. 在 DaoismCode 中输入 /paste
```

### 能力矩阵

| 场景 | 示例 |
|---|---|
| 🔍 **错误诊断** | 截图错误弹窗 → 分析原因 → 生成修复 |

**架构亮点**：视觉能力不是独立模块，而是**融入路由系统**。用户不需要手动选视觉模型——Agent 看到 `/paste` 就知道需要视觉能力，自动路由到多模态境界。

## 11. 安全模型 | Security Model

### 权限门控

所有变更工具需要用户显式批准：

```
[y] 允许一次   [n] 拒绝   [a] 全部允许   [e] 拒绝
```

### 会话加密

会话通过 **AES-256-GCM** 加密，使用从 `hostname:username` (scrypt) 派生的**机器绑定密钥**。文件在加密前 gzip 压缩。

```
文件格式: [magic 4B] [IV 12B] [authTag 16B] [ciphertext...]
```

**设计决策**：为什么用机器绑定密钥而不是用户密码？因为终端工具的用户通常不想每次启动输密码。机器绑定确保加密文件只能在同一台机器上解密——即使被复制到其他机器也无法读取。

### 路径安全

- 所有文件操作限制在项目根目录内
- `glob` 和 `read_file` 拒绝路径穿越 (`../` 转义)
- `bash` 阻止破坏性模式 (`rm -rf`, `format`) 并保护关键路径

### Git 安全网

| 机制 | 时机 | 操作 |
|---|---|---|
| **快照** | 任务首次 `edit_file` / `write_file` | `git stash create` — 捕获变更前状态 |
| **Diff守卫** | 宣布"完成"前 | `git diff --stat` — 变更过大则阻塞 |
| **回滚** | 修复次数耗尽 | 从快照恢复（手动或自动） |

### 零FFI原则

DaoismCode 不使用任何原生扩展 (node-ffi, N-API, .node 文件)。所有功能通过纯 TypeScript 实现：

| 功能 | 竞品方案 | DaoismCode 方案 |
|---|---|---|
| 终端渲染 | Ink (React) / SolidJS | 原生 ANSI 转义序列 |
| 文件监听 | chokidar (native fsevents) | 轮询 mtime |
| 加密 | node-ffi / bcrypt | Node crypto |
| 模糊搜索 | native fzf | 纯TS实现 |

这意味着：**跨平台零配置，永不崩溃于FFI冲突，单exe部署。**
